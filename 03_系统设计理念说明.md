# 优尚服饰门店运营管理系统 - 系统设计理念说明

本文档说明系统在设计阶段做出的关键决策，每个决策对应它解决的业务问题、采用的技术方案、以及在代码中的具体落点。

---

## 一、集中管控 + 门店隔离

### 解决什么问题

连锁零售场景下，总部需要统一制定商品定价和促销策略，但每家门店的库存、订单、经营数据必须独立，不能互相干扰。如果店长能看到其他门店的数据，或者门店能自行改价，会导致管理混乱。

### 设计方案

- **商品和价格全局共享**：`products` 表不绑定 store_id，所有门店看到同一份商品目录和价格
- **促销规则全局生效**：`promotions` 表无门店维度，总部配置后全店通用
- **库存按门店隔离**：`inventory` 表以 (product_id, store_id) 为联合主键，同一商品在不同门店有独立库存
- **订单归属门店**：`orders.store_id` 标记订单属于哪家店

### 代码落点

```python
# backend/app/routers/inventory.py - 门店过滤
if current_user.store_id != 0:
    query = query.filter(Inventory.store_id == current_user.store_id)
# store_id=0 是区域经理（总部），不加过滤 → 看全部门店

# backend/app/services/pos_service.py - 订单绑定当前门店
store_id = current_user.store_id
order = Order(store_id=store_id, ...)
```

区域经理的 `store_id=0` 是一个设计约定：0 代表"不属于某个门店"，查询时跳过门店过滤条件，从而实现跨店全局视图。

## 二、分级权限 RBAC（三角色 × 多权限点）

### 解决什么问题

不同岗位的人不应该看到或操作超出其职责的功能。收银员不需要看经营报表，区域经理不应该直接操作门店收银（避免误操作）。

### 设计方案

三种角色对应三种业务定位：

| 角色 | 可见功能 | 数据范围 | 业务定位 |
|------|----------|----------|----------|
| 收银员 cashier | POS收银 | 本店 | 一线操作 |
| 店长 manager | POS+库存+会员+看板 | 本店 | 门店管理 |
| 区域经理 regional | 看板+补货审批 | 全部门店 | 跨店决策 |

### 代码落点（前后端双重校验）

前端通过权限常量控制菜单渲染和路由守卫：

```javascript
// frontend/src/stores/user.js
const ROLE_PERMISSIONS = {
  cashier: ['pos'],
  manager: ['pos', 'inventory', 'members', 'dashboard', 'restock'],
  regional: ['dashboard', 'restock_approve']
}
```

后端每个需要权限的接口独立校验，不信任前端：

```python
# backend/app/routers/inventory.py
if current_user.role not in ["manager", "regional"]:
    raise HTTPException(403, "无权限")

# backend/app/routers/inventory.py - 补货审批
if current_user.role != "regional":
    raise HTTPException(403, "只有区域经理可以审批")
```

前端的权限控制负责"不让用户看到不该看的入口"（体验层），后端的权限校验负责"即使绕过前端也拿不到数据"（安全层）。两者缺一不可。

## 三、业务规则引擎化（配置字典 + JSON 规则存储）

### 解决什么问题

业务规则（会员等级阈值、促销规则类型）变化频繁。如果规则硬编码在业务逻辑中，每次调整都要改代码、重新部署。

### 设计方案

- **会员等级规则**：用 `MEMBER_LEVEL_RULES` 配置字典定义各等级的阈值、折扣率、积分倍率
- **促销规则**：用 `promotions.rules_json` 字段存储为 JSON，新增规则类型只需加一种 type 和对应的计算分支

### 代码落点

```python
# backend/app/services/pos_service.py - 会员等级配置字典
MEMBER_LEVEL_RULES = {
    "normal": {"min_spent": 0, "discount": 1.0, "points_rate": 1},
    "silver": {"min_spent": 1000, "discount": 0.98, "points_rate": 1.2},
    "gold":   {"min_spent": 5000, "discount": 0.95, "points_rate": 2},
    "diamond":{"min_spent": 15000, "discount": 0.90, "points_rate": 3},
}
```

调整等级门槛只需改这个字典，不用改任何业务逻辑代码。

```python
# 促销规则 JSON 存储示例
# 满减：{"threshold": 300, "reduction": 30}
# 品类折扣：{"category": "T恤", "discount": 0.9}

# 计算时根据 type 字段分发逻辑
if p.type == "full_reduction":
    if total_amount >= rules["threshold"]:
        promo_discount = max(promo_discount, rules["reduction"])
elif p.type == "percentage":
    cat = rules.get("category", "")
    disc = rules.get("discount", 1.0)
    if cat in category_totals:
        saving = category_totals[cat] * (1 - disc)
        promo_discount = max(promo_discount, saving)
```

要新增"买一送一"促销类型，只需：在 `promotions` 表插一条 `type="buy_one_get_one"` 的记录，在计算函数加一个 elif 分支。不改表结构，不改接口签名。

## 四、数据联动与事务一致性

### 解决什么问题

一笔收银交易涉及多张表的变更：生成订单、扣减库存、增加积分、累加消费额、判断升级。如果中途失败只改了一部分，数据就不一致了（比如扣了库存但没生成订单）。

### 设计方案

在 `create_order_logic` 中，所有变更在同一个数据库事务中完成。只有全部成功才 `db.commit()`，任一步骤抛异常则整体回滚。

### 代码落点

```python
# backend/app/services/pos_service.py - 一笔交易的完整事务
def create_order_logic(db: Session, order_data: dict, current_user):
    # 1. 验证库存 → 不足则 raise HTTPException（事务中止）
    # 2. 计算促销优惠
    # 3. 计算会员折扣
    # 4. 取最大优惠（互斥不叠加）
    # 5. 生成订单号
    # 6. 写入 orders 表
    # 7. 写入 order_items 表
    # 8. 扣减 inventory 库存
    # 9. 更新 member（积分、消费额、订单数）
    # 10. 检查等级升级
    db.commit()  # ← 以上9步全部成功才提交
    # 任何一步 raise 异常 → SQLAlchemy session 自动回滚全部变更
```

一笔交易触发 5 张表（orders、order_items、inventory、members、members.level）的同步变更，全部在一个 commit 中完成。

## 五、AI 作为管理者决策辅助工具（而非导购助手）

### 解决什么问题

门店运营中，店长面临大量决策：补多少货？经营趋势如何？哪个品类需要关注？人工分析数据效率低、容易遗漏。AI的价值是帮管理者更快、更准地做出经营决策。

### 设计原则

**定位明确：AI 服务于管理者（店长/区域经理），不服务于顾客。**

这是一个 B 端运营管理系统，不是 C 端购物 App。AI 的作用是：
- ✅ 辅助补货决策（店长在发起补货时获得数据驱动的建议）
- ✅ 经营洞察分析（店长/区域经理在看板中获得AI生成的趋势分析和建议）
- ❌ 不做导购推荐（收银员的工作是快速结账，不是向顾客推销）

**完整的降级方案：**
- AI 不参与任何核心业务计算（结账、库存变更、会员升级等）
- 调用 DeepSeek 大模型，5秒超时自动降级为规则分析
- 降级结果仍基于真实数据（日均销量、库存状态、环比变化），只是文案质量降低

### 代码落点

```python
# backend/app/routers/ai.py - 补货建议（差异化规则计算）
RESTOCK_CONFIG = {
```

## 六、逆向补偿事务（订单作废 + 等级只升不降）

### 解决什么问题

交易已完成但发现问题（录错商品、顾客退货），需要作废订单。作废时必须回滚库存和会员数据，否则数据不一致。但等级升级是"承诺"——已经告知顾客的等级不能因为作废一笔单就降回去。

### 设计方案

- 订单作废时：回滚库存、回滚积分、回滚消费额、回滚订单数
- 但不触发等级降级检查（业务规则：等级只升不降）
- 所有回滚操作加 `max(0, ...)` 保护，防止出现负数

### 代码落点

```python
# backend/app/routers/pos.py - 订单作废
@router.put("/orders/{order_id}/void")
def void_order(order_id, db, current_user):
    # 权限校验：只有 manager 和 regional 能作废
    # 状态校验：只能作废 status=completed 的订单
    # 门店校验：不能作废其他门店的订单
    
    # 回滚库存
    for item in items:
        inv.quantity += item.quantity
    
    # 回滚会员数据（但不触发降级）
    if order.member_id:
        member.points = max(0, member.points - int(order.payment_amount))
        member.total_spent = max(0, member.total_spent - order.payment_amount)
        member.total_orders = max(0, member.total_orders - 1)
        # 注意：这里故意不检查等级是否需要降级

    order.status = "voided"
    db.commit()
```

这是一个典型的"补偿事务"模式：正向操作（下单）和逆向操作（作废）是两个独立事务，逆向事务通过"反向计算"来撤销影响，但有业务约束（等级不降）的部分跳过处理。

## 总结

这套系统的设计不靠堆砌功能数量来展示复杂度，而是在有限的功能范围内，把每个功能做到"经得起追问"的程度：

1. 权限不是只做了前端菜单隐藏，而是前后端双重校验
2. 优惠不是简单打折，而是多规则互斥取最优的计算引擎
3. 数据变更不是各改各的，而是一个事务保证多表一致
4. AI 不是硬塞进来的噱头，而是服务于管理者的决策辅助工具，有完整的降级方案和明确的定位边界
5. 作废不是简单改个状态，而是完整的逆向补偿 + 业务约束保护

每一层设计都有代码可以验证，每一行代码都能回溯到它要解决的业务问题。
