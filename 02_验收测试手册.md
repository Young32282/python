### 测试目标

验证从商品搜索→加购→会员识别→优惠计算→支付→出单的完整收银闭环。重点验证优惠计算逻辑：促销满减、品类折扣、会员折扣三者取最优（不叠加）。

### 前置条件

- 使用 `cashier1` / `123456` 登录
- 系统当前有生效的促销：满300减30（full_reduction）、T恤9折（percentage）

### 📌 测试会员（固定手机号，每次重置数据后不变）

| 手机号 | 姓名 | 等级 | 累计消费 | 用途 |
|--------|------|------|----------|------|
| **13800001001** | 测试银卡 | silver | ¥2500 | 本用例验证会员折扣 |

### 操作步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在搜索框输入“T恤”，等待搜索结果 | 下拉列表显示多个 T 恤类商品，每个条目显示商品名、价格和库存 |
| 2 | 点击其中一个商品（如“T恤经典款01”） | 商品被添加到购物车表格，数量为 1，底部显示合计金额 |
| 3 | 在购物车中将该商品数量改为 2 | 小计金额 = 单价 × 2，合计金额同步更新 |
| 4 | 在“会员识别”区域，输入 **13800001001**，点击【识别】 | 显示“测试银卡”，等级 silver，积分 1500，折扣 98折 |
| 5 | 观察“优惠信息”卡片 | 显示原价、优惠金额、实付金额。系统自动比较满减/品类折扣/会员折扣，取最大优惠 |
| 6 | 支付方式选择“微信”（默认已选），点击【结账】 | 弹出“结账成功”对话框，显示订单号（格式 ORDyyyyMMddXXXX）和实付金额 |
| 7 | 点击对话框【确定】 | 对话框关闭，购物车清空，会员信息清空 |

### 优惠计算逻辑说明

```
原价 total_amount = Σ(单价 × 数量)

促销优惠（取最大值）：
  - 满减：total_amount >= 300 → 减30
  - 品类折扣：T恤品类金额 × (1 - 0.9) = T恤金额 × 10%

会员折扣：
  - 银卡 98 折：total_amount × (1 - 0.98) = total_amount × 2%

最终优惠 = max(促销优惠, 会员折扣)  ← 不叠加，取最优
实付金额 = 原价 - 最终优惠
```

### 数据验证点

| 数据表 | 验证内容 |
|--------|----------|
| `orders` | 新增一条记录，order_no 格式正确，store_id=1，cashier_id=1，status=completed |
| `order_items` | 新增对应商品明细，unit_price 和 quantity 正确 |
| `inventory` | 对应商品 store_id=1 的库存减少了购买数量 |
| `members` | 该会员 points 增加（实付金额 × 1.2 银卡倍率），total_spent 增加 |

### 操作步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 用 cashier1 登录 POS，搜索商品加入购物车，输入 **13800001003** 识别会员 | 显示“测试回滚”，等级 gold |
| 2 | 记录当前会员数据：points=5000, total_spent=5200, level=gold | 用于回滚后对比 |
| 3 | 点击【结账】，记录订单号和实付金额 | 结账成功，记下订单号 |
| 4 | 用 `manager1` / `123456` 登录，进入 POS 页面查看今日订单 | 能看到刚才的订单，状态为 completed |
| 5 | 点击该订单的【作废】按钮 | 确认后提示“订单已作废” |
| 6 | 验证数据库中会员数据 | 订单 status=voided，会员 points 和 total_spent 回滚，但 **level 仍为 gold** |

### 数据验证点

| 数据表 | 验证内容 |
|--------|----------|
| `orders` | status 从 completed → voided |
| `inventory` | 商品库存恢复（+购买数量） |
| `members.points` | 积分回滚（减去该笔订单获得的积分，最小为0） |
| `members.total_spent` | 消费额回滚（减去该笔订单的支付金额，最小为0） |
| `members.total_orders` | 订单数 -1 |
| `members.level` | **保持不变**（只升不降，即使 total_spent 已低于当前等级阈值） |

### 逆向补偿逻辑说明

```
作废订单触发：
  1. 库存回补：inventory.quantity += order_item.quantity
  2. 积分回滚：member.points = max(0, points - earned_points)
  3. 消费额回滚：member.total_spent = max(0, total_spent - payment_amount)
  4. 订单数回滚：member.total_orders = max(0, total_orders - 1)
  5. 等级不动：不检查降级条件（业务规则：等级只升不降）
```

## TC-6：AI 辅助决策（经营洞察 + 补货建议）

### 测试目标

验证AI作为管理者决策辅助工具的两个功能：
1. 经营洞察（Dashboard）：调用DeepSeek分析销售和库存数据，给出经营建议
2. 补货建议（库存管理）：基于销售数据的差异化规则计算

### AI在系统中的定位

本系统是B端运营管理工具，AI服务于**管理者（店长/区域经理）**而非顾客：
- ✅ 经营洞察：帮助管理者理解销售趋势、发现异常、获得可执行建议
- ✅ 补货建议：帮助店长在补货时做出数据驱动的决策
- ❌ 不做导购推荐（收银员的职责是快速结账，不是向顾客推销）

### 前置条件

- `config.py` 中 `AI_API_KEY` 已配置真实 DeepSeek API Key
- 使用 manager1 或 regional1 账号（仅管理者可访问）
- 确保网络可访问 api.deepseek.com

### 操作步骤（经营洞察 - Dashboard）

| 步骤 | 操作 | 预期结果 |

| 1 | 用 `manager1` / `123456` 登录，进入经营看板 | 看板页面加载，底部“AI经营洞察”卡片自动加载 |

### AI经营洞察 Prompt

```
System: 你是零售门店的经营分析师。根据提供的销售和库存数据，给出简洁的经营建议。
要求：1.基于数据 2.可执行 3.3-5条 4.用数字说话

User: 近7天经营数据：
- 销售总额：¥{X}（环比增长/下降{Y}%）
- 订单数：{N}单，客单价：¥{Z}
- 库存预警：{W}件低于安全线，{S}件缺货
- 品类销售：T恤(¥A)、衬衫(¥B)...

```

### 补货建议差异化策略

| 商品类型 | 日均销量 | 安全周期 | 文案标识 |
